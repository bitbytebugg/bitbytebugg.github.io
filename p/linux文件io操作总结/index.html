Linux文件IO操作总结

Aug 02, 2024
3 minute read

Linux API

Linux中所有执行IO操作的系统调用都围绕着文件描述符来进行，文件描述符是一个非负整数，来指代打开的文件。这里"打开的文件"包括管道、FIFO、socket、终端、设备和普通文件。针对每个进程，文件描述符都自成一套：

  • #include <unistd.h>
  • STDIN_FILENO 标准输入流的文件描述符 0
  • STDOUT_FILENO 标准输出流的文件描述符 1
  • STDERR_FILENO 标准错误流的文件描述符 2
  • fd = open(pathname, flags, mode)
  • numread = read(fd, buffer, count) 如何读到文件结束符EOF（Linux终端为CTRL+D），会返回0
  • numwritten = write(fd, buffer, count)
  • status = close(fd)

C标准库API

C标准库中所有执行IO操作的库函数围绕文件指针 FILE*来进行：

  • #include <stdio.h>
  • stdin 标准输入流的文件指针
  • stdout 标准输出流的文件指针
  • stderr 标准错误流的文件指针
  • FILE* = fopen(filepath, mode)
  • rnum = fread(buf, 5,/*每个数据的字节数*/ 2,/*数据的个数*/ FILE* stream);
  • wnum = fwrite(buf, 5, 2, FILE* stream);
  • fclose()

对比

  • 是否带缓冲：
    • 操作文件描述符的Linux API默认无缓冲（除非手动设置，如 setbuf() 或使用 read()/write() 结合缓冲区），适合需要精细控制 I/O 的场景（如网络编程、底层设备操作）。
    • 操作文件指针的C API 默认带缓冲（全缓冲、行缓冲或无缓冲，可通过 setvbuf() 设置）。缓冲机制减少系统调用次数，提高效率（例如 fprintf() 先写入缓冲区，满后再调用 write()）。
  • 有无缓冲造成性能和其他方面影响：
    • 每次 read()/write() 都会触发系统调用，频繁操作时开销较大。适合小数据量或需要实时响应的场景。
    • fread()/fwrite() 缓冲机制减少系统调用，适合批量数据读写（如文件解析、日志写入）。但缓冲可能导致数据延迟写入（如需立即写入，需调用 fflush()）。
  • 功能差异
    • 操作文件描述符的Linux API 支持更多底层操作（如 fcntl() 设置非阻塞 I/O、ioctl() 设备控制）。可用于操作非文件类资源（如套接字、管道、设备文件）。
    • 操作FILE* 的C API 提供更丰富的格式化 I/O 函数（如 fprintf()、fscanf()）。不支持直接操作套接字或设备（需先转换为文件描述符 fileno()）。

如何理解open write之类Linux API是无缓冲的：

在Linux系统编程中，open()和write()等系统调用通常被称为"无缓冲"(unbuffered)的I/O操作，这是与标准I/O库函数(如fopen(), fwrite())相比较而言的。以下是详细解释：

  1. 直接系统调用：
    • open()和write()是直接与内核交互的系统调用
    • 每次调用都会直接进入内核空间，不经过用户空间的缓冲层
  2. 无用户空间缓冲：
    • 标准I/O库(如glibc)会在用户空间维护缓冲区
    • 而系统调用级别的I/O不提供这种缓冲机制

文件描述符和文件指针的互相转换：

  • fileno(FILE*)：从 FILE* 获取文件描述符。
  • fdopen(int fd, const char *mode)：从文件描述符创建 FILE*。

线程安全性：

  • read/write等系统调用本身是原子操作，但多线程操作同一 fd 需额外同步。
  • C 标准库函数（如 fprintf()）通常是线程安全的（内部有锁机制）。

C++ API

C++标准库提供三种类，实现C++对文件的操作：

  • include <fstream>
  • ofstream 写文件类，由 ostream 引申而来
  • ifstream 读文件类，由 istream 引申而来
  • fstream 同时读写文件类，由iostream 引申而来
