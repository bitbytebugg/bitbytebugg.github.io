格式概览

整个ELF文件可以分成四块：

┌───────────────────────┐
│      ELF File Header  │  ► 描述文件类型、架构、入口点等全局信息
├───────────────────────┤
│    Program Headers    │  ► 定义运行时内存布局（Segments）
│ (Optional for reloc.) │
├───────────────────────┤
│       Sections        │  ► 实际存储代码、数据等（如 .text, .data）
│                       │
├───────────────────────┤
│    Section Headers    │  ► 描述各节的逻辑信息（供链接器使用）
│ (Optional for exec.)  │
└───────────────────────┘

ELF格式本质上是一些称为 Headers 的较小数据结构的集合。一个 Header 是由一些更简单（或原始的 primitive）的数据类型（例如整数）组成的连续序列。有多种类型的 Headers ，每种类型描述一种信息。下面这张图描述了ELF中包含的Header以及组成Header的原始数据类型之间的关系。

原始数据类型

原始数据类型（primitive data type）是编程语言中最基本的数据类型，具有如下特点：

  • 原始值一旦创建就不能被修改
  • 在赋值或传递给函数时，传递的是值的副本
  • 值直接存储在变量访问的位置

ELF定义了几种原始类型，ELF格式中定义的其他任何结构都是这几种原始类型的组合：

typedef uint64_t Elf64_Addr;        Memory address: 64 bit unsigned integer
typedef uint64_t Elf64_Off;         Offset: 64 bit unsigned integer

ELF Header

整个ELF Header的大小为64字节。

typedef struct elf64_hdr {
// 16字节长度
  unsigned char	e_ident[EI_NIDENT];
  Elf64_Half e_type;
  Elf64_Half e_machine;
  Elf64_Word e_version;
  Elf64_Addr e_entry;		/* Entry point virtual address */
  Elf64_Off e_phoff;		/* Program header table file offset */
  Elf64_Off e_shoff;		/* Section header table file offset */
  Elf64_Word e_flags;
  Elf64_Half e_ehsize;
  Elf64_Half e_phentsize;
  Elf64_Half e_phnum;
  Elf64_Half e_shentsize;
  Elf64_Half e_shnum;
  Elf64_Half e_shstrndx;
} Elf64_Ehdr;

e_ident

ident是identity的缩写，意味ELF文件的身份信息。

通过宏 EI_NIDENT写死了是16个元素的字节数组。通过 elfcat 结合 man elf 可以了解到这16个字节包含的信息：

  • 4字节：\x7f E L F
  • 1字节：32位/64位
  • 1字节：小端/大端，Linux+GCC生成的ELF文件默认为小端
  • 1字节：EI_VERSION EFL规范的版本信息，必须为1，其他值无效
  • 1字节：EI_OSABI 操作系统ABI信息
  • 1字节：EI_ABIVERSION
  • 当前规范中后面7个字节为padding（全0），后面的规范可以会变化

e_type

2个字节表示对象文件类型：

  • 未知
  • 可重定位文件
  • 执行文件
  • 共享对象文件
  • core file

e_machine

2个字节指明目标架构：

  • 00 3e AMD x86-64

e_entry

8个字节（刚好是64位指针大小）表示程序入口点的虚拟地址（vaddr）。如果为0表示该文件没有关联入口点。

e_phoff

8个字节表示**Program headers (Program Header Table)**在本文件中的字节偏移量。如果没有程序头表，该值为0

e_shoff

8个字节表示**Section Headers (Section Header Table)**在本文件中的字节偏移量。如果没有节头表，该值为0

e_ehsize

2个字节表示ELF Header的大小

e_phentsize

2个字节表示Program Header Table中一个entry的字节大小，注意Program Header Table中每个entry的大小都是相同的。

e_phnum

2个字节表示Program Header Table中entry的数量，注意 e_phentsize * e_phnum 等于整个Program Header Table的字节大小。

e_shentsize

2个字节表示Section Header Table中一个entry的字节大小，Section Header Table中一个entry就是一个Section Header，所有的entry的大小都是一样的。

e_shnum

2个字节表示Section Header Table中entry的数量，注意 e_shentsize * e_shnum 等于整个Section Header Table的字节大小。

e_shstrndx

2个字节表示 .shstrtab Section Header在整个Section Header Table中的索引。.shstrtab 是 Section Header Table中的一个entry，也就是一个Section Header，该 Section Header 是一个字符串表（String Table）里面保存了所有Section Header的名字。

Program Header Table

程序头表只在ELF可执行文件和动态共享对象中存在，因为程序头表描述的是所有Section合并成Segment的结果，只有能被加载执行的才具有该项目。

程序头表紧跟着ELF Header从ELF文件的第65个字节开始，从C语言代码的角度来看，可以看作是一个结构体数组。这里结构体 Elf64_Phdr 表示的就是程序头表中的一个 Program Header（在一些场景中Program Header也被叫做Segment；同样Section Header在一些场景中也被叫做Section）：

p_type

4个字节表示程序头表中的元素（Segment / Program Header）的类型。该类型用于指导怎么解析程序头数组元素的信息：

  • PT_NULL 表示数组元素是未使用的，且元素的成员也是未定义的，该标志用于表明忽略该元素。
  • PT_LOAD 该数组元素表示一个可加载的Segment
  • PT_DYNAMIC 该数组元素表示动态链接信息
  • PT_INTERP 该数组元素指定了一个以空字符结尾的路径名的位置和大小，该路径名用于指定要调用的解释器。此段类型仅对可执行文件有效（尽管它可能出现在共享对象中），但一个文件中最多只能出现一次。如果存在，它必须位于任何可加载Segment条目之前。
  • PT_NOTE
  • PT_SHLIB 该类型是保留的，目前未明确定义其语义。若程序包含此类数组元素，则不符合ABI规范。
  • PT_PHDR 若存在该数组元素，则其用于指明程序头表（program header table）在文件及程序内存映像中的位置和大小。此段类型在文件中最多只能出现一次，且仅当程序头表属于程序内存映像的一部分时才允许存在。若存在，则其必须位于所有可加载段条目之前。
  • PT_LOPROC, PT_HIPROC 位于 [PT_LOPROC, PT_HIPROC] 之间的值是为特定处理器语义而保留的。
  • PT_GNU_STACK GNU扩展功能，该功能被Linux内核用于通过p_flags成员中设置的标志位来控制堆栈状态。

p_offset

该成员存储了Segment首字节在ELF文件中的起始偏移量（单位字节）。

p_vaddr

该成员记录了该Segment首字节在内存中的虚拟地址。

p_paddr

该字段在采用物理寻址的系统上是有意义的，此成员被保留用于存储段的物理地址。在BSD系统下，该成员未被使用且必须置零。

p_filesz

表示该Segment在ELF文件映像（磁盘上的二进制表示）中的字节数大小，可能为0

p_memsz

表示该Segment在内存映像（内存中的二进制表示）中的字节数大小，可能为0

p_flags

该成员存储了与该Segment相关的标志位掩码：

  • PF_X 一个可执行的Segment
  • PF_W 一个可写的Segment
  • PF_R 一个可读的Segment 举例：一个 text segment通常拥有 PF_X和PF_R。一个 data segment通常拥有 PF_W和PF_R。

p_align

该成员定义了Segment在内存及文件中的对齐值。可加载进程段必须满足 p_vaddr 与 p_offset 在页面大小模数下的对齐一致性。若该值为零或一，则表示无需对齐；否则，p_align 应为 2 的正整数次幂，且 p_vaddr 需等于 p_offset，而且 p_vaddr/p_offset 对 p_align 取模的余数均为0。

/// linux/include/elf.h

typedef struct
{
  Elf64_Word	sh_name;		/* Section name (string tbl index) */
  Elf64_Word	sh_type;		/* Section type */
  Elf64_Xword	sh_flags;		/* Section flags */
  Elf64_Addr	sh_addr;		/* Section virtual addr at execution */
  Elf64_Off	    sh_offset;		/* Section file offset */
  Elf64_Xword	sh_size;		/* Section size in bytes */
  Elf64_Word	sh_link;		/* Link to another section */
  Elf64_Word	sh_info;		/* Additional section information */
  Elf64_Xword	sh_addralign;	/* Section alignment */
  Elf64_Xword	sh_entsize;		/* Entry size if section holds table */
} Elf64_Shdr;

Section Header Table的部分索引值是被保留的（ELF文件不会分配这些Section Header）：

  • 一是初始条目（索引位置为0），在ELF扩展的情况下用于存储 e_phnum、e_shnum和e_shstrndx的值；在其他情况下各字段均为零。
  • 二是 [SHN_LORESERVE, SHN_HIRESERVE] （索引位置为 [0xff00, 0xffff]）

sh_name

4个字节表示 .shstrtab Section （Section Header String Table Section）中的索引，该索引位置元素是 Section的名字（这个字符串是C风格的以\0结尾的字符串）。

sh_type

4个字节，用于表示该Section Header对应的Section的类型：

  • SHT_NULL 这个值表示该节头（section header）是非活跃的，它没有关联的实际节区（section）。节头中的其他成员的值是未定义的。
  • SHT_PROGBITS 该Section包含由程序自身定义的信息，其格式和含义完全由程序决定。
  • SHT_SYMTAB 该Section包含一个符号表。通常情况下，SHT_SYMTAB 类型的符号表主要用于链接编辑（link editing），但也可能被动态链接（dynamic linking）使用。由于它是一个完整的符号表，可能会包含许多动态链接所不需要的符号。此外，目标文件（object file）还可能包含一个 SHT_DYNSYM 节区，专门用于动态链接。
  • SHT_STRTAB 该Section包含一个字符串表(String Table)。一个ELF文件可能拥有多个字符串表Section
  • SHT_RELA 该Section存储了带有显式加数（explicit addends）的重定位项，例如 32 位目标文件中的 Elf32_Rela 类型。一个目标文件可能包含多个重定位节区。
  • SHT_HASH 该Section存储了一个符号hash表。任何参与动态链接（dynamic linking）的目标文件或共享库必须包含一个符号哈希表，以加速运行时符号查找（如通过 dlopen 或 dlsym）。若缺失此表，动态链接器（ld.so）将无法高效解析符号，导致性能下降或错误。每个目标文件只能包含一个哈希表（通常位于 .hash 或 .gnu.hash 节区）。
  • SHT_DYNAMIC 该Section存储了动态链接信息。一个ELF文件只能有一个 dynamic section。（may have only one 此处 “may” 不是表示"可能"，而是表示规范中的允许性限制（类似"shall"但语气较弱），应译为"只能"或"仅允许"。技术文档中常见的 “may only” 结构均表示强制性限制（如 RFC 2119 标准定义）。
  • SHT_NOTE 该Section包含 notes (ElfN_Nhdr)
  • SHT_NOBITS 此类节区（section）在文件中不占用实际空间，但其他特征与 SHT_PROGBITS 类型节区类似。尽管该节区不包含实际数据字节，其 sh_offset 成员仍保留逻辑文件偏移量的概念值。
  • SHT_REL 该Section存储的是无显式加数（explicit addends）的重定位偏移量，例如32位目标文件中的Elf32_Rel类型。一个目标文件中可能包含多个重定位节区。
  • SHT_SHLIB 该类型保留（目前没有确定的语义）
  • SHT_DYNSYM 该Section存储的是动态链接所需的最简符号集合。目标文件可能同时包含一个SHT_SYMTAB节区（完整符号表）。
  • SHT_LOPROC, SHT_HIPROC
  • SHT_LOUSER
  • SHT_HIUSER

sh_flags

8个字节，每个bit为一个标志位：

  • SHF_WRITE 该Section包含内容在进程执行过程中是可写的
  • SHF_ALLOC 该Section在进程执行期间会占用内存空间。某些控制节区不会驻留在目标文件的内存映像中，此类节区将禁用此属性。
  • SHF_EXECINSTR 该Section包含可执行的机器指令
  • SHF_MASKPROC 该掩码包含的所有bit是为处理器特定语义保留的。

sh_addr

8个字节，若该Section存在于进程的内存映像中，则此成员（sh_addr）存储节区首字节应驻留的内存地址；否则，该成员值为零。

sh_offset

8字节，该成员（sh_offset）的值表示从文件起始位置到该Section首个字节的字节偏移量。特别地，对于SHT_NOBITS类型的节区（如.bss），其虽然在文件中不占用实际存储空间，但sh_offset成员仍会标识该节区在文件中的逻辑位置。

sh_size

8字节，表示该Section的大小（单位字节）

sh_link

4个字节，Section的索引号，指向当前Section关联的其他Section，具体含义取决于当前Section类型（sh_type）

sh_link和sh_info两个字段表示段的链接信息，如果段的类型是与链接相关的（不论是动态链接还是静态链接），比如重定位表、符号表这些段，那么 sh_link 和 sh_info 这两个成员所包含的意义如下表所示（对于其他类型的段，这两个成员没有意义）：

sh_type sh_link sh_info
SHT_DYNAMIC 该段所使用的字符串表在section header table中的索引 0
SHT_HASH 该段所使用的符号表在section header table中的索引 0
SHT_REL 该段所使用的相应符号表在section header table中的索引 该重定位表所作用的Section在section header table中的下标
SHT_RELA 该段所使用的相应符号表在section header table中的索引 该重定位表所作用的Section在section header table中的下标
SHT_SYMTAB 操作系统相关的 操作系统相关的
SHT_DYNSYM 操作系统相关的 操作系统相关的
other SHN_UNDEF 0

sh_info

4个字节，表示额外信息，具体语义由 sh_type 决定

sh_addralign

8个字节。某些节区具有地址对齐约束。例如，若某节区存储的是双字（doubleword）数据，则系统必须确保整个节区按双字对齐。具体表现为：该节区的 sh_addr 地址值必须满足 sh_addr ≡ 0 (mod sh_addralign) 的同余条件。对齐值仅允许为 0、1 或 2 的正整数幂次方——当取值为 0 或 1 时，表示该节区无对齐要求。

sh_entsize

部分节区存储的是由固定长度条目构成的表结构（如符号表）。对于此类节区，本字段（sh_entsize）表示每个条目所占的字节大小；若节区内容非固定长度条目表，则本字段值为零。

Sections

关于Section的名字，下面列出的Section的名字都是以 . 开头的，表示这些Section名字都是系统保留的，应用程序也可以使用一些非系统保留的名字作为Section名字。自定义的Section名字不能使用 . 开头，以免和系统保留名字冲突。ELF文件中也可以拥有几个相同段名的Section。

.bss

存储的是未初始化的全局变量和局部静态变量，这些数据会占用程序的内存映像空间。根据定义，系统会在程序启动运行时将其内容初始化为零值。此节区类型为 SHT_NOBITS（不占用文件空间），其属性标志包含 SHF_ALLOC（需分配内存）和 SHF_WRITE（可写）。

注意.bss段在文件中不占用实际存储空间（仅定义未初始化的全局/静态变量）。也就是说在ELF文件中.bss段只有元数据（如sh_size定义大小），无实际内容，而且其对应节区头（Section Header）描述中的sh_offset字段通常为0或无效值。

.ctors 和 .dtors

.ctors Section存储的是已初始化的C++构造函数指针，其类型为SHT_PROGBITS（包含程序定义的已初始化数据），并具有SHF_ALLOC（需加载到内存）和SHF_WRITE（可写）属性标志。

.dtors Section存储的是已初始化的C++析构函数指针，其类型为SHT_PROGBITS（包含程序定义的已初始化数据），并具有SHF_ALLOC（需加载到内存）和SHF_WRITE（可写）属性标志。

.data

该Section存储的是已初始化的程序数据，这些数据将构成进程的内存映像。其类型为SHT_PROGBITS（包含实际初始数据内容），并具有SHF_ALLOC（需分配内存）和SHF_WRITE（可写）属性标志。.data Section保存了初始化了的全局变量和局部静态变量。

.rodata

存放只读数据，一般是程序里面的const修饰的变量和字符串变量。有些编译器会把字符串变量放到 .data Section。

.rel.text (32位)

ELF文件中 .rel.text 和 .rela.text 都是与代码段 .text 相关的重定位Section。

  • .rel.text
    • 类型 Elf32_Rel/Elf64_Rel
    • 加数Addend隐式存储在需要重定位的目标位置
    • 一般使用在较旧的工具链或者简单架构里
  • .rela.text
    • 类型 Elf32_Rela/Elf64_Rela
    • 加数Addend显式给出，条目中直接包含 r_addend 字段
    • 一般使用在现代主流架构里（如x86-64）

一句话总结：Rel 用在32位系统里，Rela则用在64位系统里。

objdump -r 用来查看 main.o 中所有要重定位的地方，即 main.o 所有引用到外部符号的地址。每个要被重定位的地方叫一个重定位条目（Relocation Entry）, 每个条目对应一个结构体 Elf64_Rela ，该结构体字段：

  • r_offset：表示该重定位需要修正的内容的第一个字节相对于该符号表关联的Section（比如 .text）段起始位置的偏移量（单位字节）
  • r_info：该成员（r_info）同时包含需执行重定位所依据的符号表索引和待应用的重定位类型。重定位类型的具体定义与处理器架构相关。当文档提及重定位条目的"重定位类型"或"符号表索引"时，即表示分别对条目的r_info成员应用ELF[32|64]_R_TYPE或ELF[32|64]_R_SYM宏所提取的结果。部分字节表示类型，部分字节表示符号表索引，elf.h 有对应宏提取各部分的信息。
  • r_addend：对于结构体Elf64_Rel缺少的这个字段，其实是隐式写在 r_offset 指向的位置。这个值加到哪里？猜测是指令修正中用到的一个常数。

对于所有在 readelf -s main.o 中 Bind列显示为 GLOBAL 且 Ndx 列显示UND (undefined) 的符号，在重定位表中都应该至少有一项（每个重定位符号在读取和写入时都应该有一个 Relocation Entry）。

重定位的本质是指令修正

重定位本质是指令修正，牢记这一点对于理解重定位概念和实现逻辑至关重要。ELF定义了32种不同的重定位类型，这里主要讲解其中两种最基本的：

  • R_X86_64_PC32。重定位一个使用 32 位 PC 相对地址的引用。回顾《程序员的自我修养》3.6.3 节，一个 PC 相对地址就是距程序计数器（PC）的当前运行时值的偏移量。当 CPU 执行一条使用 PC 相对寻址的指令时，它就将在指令中编码的 32 位值加上 PC 的当前运行时值，得到有效地址（如 call 指令的目标），PC 值通常是下一条指令在内存中的地址。
  • R_X86_64_32。重定位一个使用 32 位绝对地址的引用。通过绝对寻址，CPU 直接使用在指令中编码的 32 位值作为有效地址，不需要进一步修改。 这两种重定位类型支持 x86-64 小型代码模型（small code model），该模型假设可执行目标文件中的代码和数据的总体大小小于 2GB，因此在运行时可以用 32 位 PC 相对地址来访问。GCC 默认使用小型代码模型。大于 2GB 的程序可以用 -mcmodel=medium（中型代码模型）和 -mcmodel=large（大型代码模型）标志来编译，不过在此我们不讨论这些模型。

字符串表 .strtab 和 .shstrtab

字符串表将所有字符串（段名，变量名，函数名）接连排列成线性存储，因为采用的是C风格字符串（yi \0 结束），所以只需要给定字符串起始位置的索引，就可以访问这个字符串的内容了。

.rela.text 是针对 .text 段的重定位表，当 .text 中至少有一个绝对地址的引用（比如调用当前源文件之外定义的函数例如 printf 库函数）。所谓的重定位表其实就是ELF文件中的一个Section，而每个Section都有一个Section Header。重定位表的Section Header中 sh_link 表示符号表的下标，sh_info 表示它作用于哪个Section。

符号表 .symtab

每一个目标文件都会有一个相应的符号表，这个表里面记录了目标文件中所用到的所有符号。每个定义的符号有一个对应的值，叫做符号值，对于变量和函数来说，符号值就是他们的地址。除了函数和变量之外，还存在其他几种不常用到的符号。我们将符号表中的所有符号进行分类，它们可能是下面这些类型中的一种：

  • 全局符号：定义在一个目标文件中的全局函数和全局变量，它们可以被其他目标文件引用。
  • 外部符号：在本目标文件中引用的全局符号，却没有定义在本目标文件中。
  • 段名：这种符号一般由编译器生成，它的值就是该段的起始地址。
  • 局部符号：这类符号只在编译单元内部可见。比如static修饰的变量。调试器可以使用这些符号来分析程序或崩溃时的coredump文件。这些局部符号对于链接过程没有作用，连接器往往也忽略它们。
  • 行号信息：即目标文件指令与源代码中代码行的对应关系，它也是可选的。

// ELF 符号表结构
/// linux/include/elf.h

typedef struct {
  Elf64_Word	st_name;		/* Symbol name (string tbl index) */
  unsigned char	st_info;		/* Symbol type and binding */
  unsigned char st_other;		/* Symbol visibility */
  Elf64_Section	st_shndx;		/* Section index */
  Elf64_Addr	st_value;		/* Symbol value */
  Elf64_Xword	st_size;		/* Symbol size */
} Elf64_Sym;
  • st_name 表示该符号名在字符串表 .strtab 中的索引
  • st_value 符号相对应的值。这个值跟符号有关，可能是一个绝对值，也可能是一个地址等，不同的符号，它所对应的值含义不同。
    • 目标文件中：
      • 如果符号不是COMMON块类型（st_shndx != SHN_COMMON ），st_value 表示该符号在段中的偏移。即符号位于由 st_shndx 指定的段，偏移为 st_value 的位置。
      • 如果符号是COMMON块类型，st_value 表示该符号的对齐属性。
    • 可执行文件中：
      • st_value 表示符号的虚拟地址。这个虚拟地址对动态链接器非常有用。关键词 “动态链接器”
  • st_size 符号大小，对于包含数据的符号，这个值是该数据类型的大小。比如一个double型的符号它占用8个字节。如果值为0，表示符号大小为0或未知。
  • st_info 一个字节表示符号类型和绑定信息，低4位表示符号类型，高4位标识绑定信息。
    • STT_NONE 未知类型符号
    • STT_OBJECT 类型为数据对象，比如变量、数组
    • STT_FUNC 类型为函数或其他可执行代码
    • STT_SECTION 类型为一个段，对应的绑定信息必须为STB_LOCAL
    • STT_FILE 类型为一个文件名，一般为目标文件对应的源文件名，对应的绑定信息也一定是STB_LOCAL的，而且它的st_shndx一定是 SHN_ABS (0xfff1) 标识该符号是一个绝对符号。
  • st_other 该成员目前为0，没用
  • st_shndx 符号表中定义的每一项都是和一个Section相关的，该字段表明与之相关的 Section 的信息。
    • 如果符号定义在本目标文件中，那么这个成员标识符号所在的Section对应的Section Header在Section Header Table中的索引。
    • 如果符号不是定义在本目标文件中，或者某些特殊符号，sh_shndx会有特殊值：
      • SHN_ABS = 0xfff1 表示该符号包含了一个绝对的值，比如表示文件名的符号就属于这种类型
      • SHN_COMMON = 0xfff2 表示该符号是一个COMMON块类型的符号，一般来说，未初始化的全局符号定义就是这种类型的，详细参考 “静态链接 COMMON块”
      • SHN_UNDEF = 0 表示该符号未定义，这个符号在本目标文件被引用到，但是定义在其他目标文件中。

通过 readelf -s 查看符号表，输出的含义：

  • Num：表示该条目在符号表数组中的下标
  • Value：表示符号值，就是 st_value
  • Size：表示符号大小，对应st_size
  • Type：对应st_info的符号类型信息
  • Bind：对应 st_info 的符号绑定信息
  • Vis：目前在C/C++中未使用，可以忽略它
  • Ndx：对应 st_shndx
  • Name：符号名称，对应 st_name，已从字符串表中解析出来。 从输出中可以看到，第一个符号，即下标为0的符号，永远是一个未定义的符号。
