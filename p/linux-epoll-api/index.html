理解水平触发和边缘触发

select()和poll()是采用水平触发(Level Triggered)，也就是这两接口会将就绪的文件描述符告诉进程，如果进程没有对这些就绪文件描述符进行IO处理，那么下次调用 select() 和 poll() 的时候还会再次报告这些文件描述符，**这里把未就绪状态比作低电平，就绪状态比作高电平，那么水平触发的含义是只要文件描述符处于高电平就会持续报告这个文件描述符需要被处理。

epoll()可以同时支持水平触发和边缘触发(Edge Triggered)。边缘触发的含义是对于哪些文件描述符处于就绪状态，epoll()接口只会报告一遍，如果我们没有对其进行处理，那么epoll()下次将不会再次报告。**这里还是把未就绪状态比作低电平，就绪状态比作高电平，那么边缘触发的含义是只有在文件描述符从低电平变为高电平的那个瞬间才会报告这个文件描述符，过了这个电平变化的时刻，即使处于高电平状态也不再报告这个文件描述符了。理论上边缘触发的性能要更高一些，但是代码实现相当复杂。

select()和poll()经常放在一起指代，select()和poll()的区别可以看这里 select和poll的区别。结论就是有小的差别，但是select()存在的主要缺点，poll()也有。后续无特殊说明时，select()就代指select()和poll()这两个接口。

fd_set read_fds;
FD_ZERO(&read_fds);
FD_SET(fd1, &read_fds);
FD_SET(fd2, &read_fds);
select(max_fd+1, &read_fds, NULL, NULL, NULL);

// 必须遍历所有fd判断哪些就绪
for (int i = 0; i <= fd_list.size(); i++) {
    fd_i = fd_list[i];
    if (FD_ISSET(fd_i, &read_fds)) {  // 逐个检查
        // 处理可读的fd_i

    }
}

fd_set 是一个用于I/O多路复用（select()，pselect()等系统调用）的数据结构，主要用于高效管理文件描述符的集合。

  • 实现本质：fd_set 本质上是一个位数组，每个比特对应表示一个文件描述符是否在 fd_set 中。
  • 固定大小：传统实现中，fd_set 的大小由常量 FD_SETSIZE 决定（通常为 1024，依赖系统配置）。这意味着默认情况下，select() 只能监控 fd 值 < FD_SETSIZE 的文件描述符。
  • 配套有对应的操作宏：提供了一组宏对 fd_set 进行操作：
    • FD_ZERO(set)：清空集合。
    • FD_SET(fd, set)：将 fd 加入集合。
    • FD_CLR(fd, set)：从集合移除 fd。
    • FD_ISSET(fd, set)：检查 fd 是否在集合中。

从上面的模式代码中，我们可以总结出 select() 的工作方式存在以下特点（缺点）：

  1. 用户传递全量fd集合（如1024个fd的位图）给内核。
  2. 内核线性扫描所有fd，检查它们的可读/可写状态。
  3. 内核修改用户传入的fd集合，将就绪的fd对应位设为1，其余清零。
  4. 返回全量的fd（而非告诉我们具体哪些fd是就绪的）。
  5. 用户需重新遍历所有fd（即使大部分未就绪）通过 FD_ISSET 判断哪些可用。这里的时间复杂度就是 $O(n)$ ，其中n为监控的fd总数。

/*
  使用 epoll API 在用户空间的代码模板示例
*/

int epfd = epoll_create1(0);

// 创建一个能存放多个就绪事件的数组。这里数组长度 10 表示 epoll_wait() 一次最多返回 10 个就绪事件。这个可以根据需求调整。越大的数组可以避免频繁调用 epoll_wait()；越小的数组单次处理的事件量越少，利于降低延迟。
struct epoll_event events[10];

// 注册事件到epoll上，可以多次调用来注册多个事件
struct epoll_event ev;
ev.events = EPOLLIN;
ev.data.fd = a_fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, a_fd, &ev);
...

// 等待事件，返回n为就绪事件个数
int n = epoll_wait(epfd, events, 10, -1);

// 直接处理返回的就绪事件（无需遍历）
// 如果 n > 10，则需要继续调用 epoll_wait()
for (int i = 0; i < n && i < 10; i++) {
    int ready_fd = events[i].data.fd;  // 直接获取就绪fd
    // 处理可读的fd
}

epoll_create1(0)是linux的系统调用，用于创建一个新的 epoll 实例。它是 epoll_create 的改进版本，提供了更灵活的控制选项：

  • 参数0：表示表示调用 epoll_create1 时不设置任何标志（flags）。目前唯一支持的标志是： EPOLL_CLOEXEC，如果设置了这个标志（如 epoll_create1(EPOLL_CLOEXEC)），文件描述符会在 exec 系列函数调用时自动关闭，避免子进程继承。
  • 返回值：返回一个新的 epoll 文件描述符（非负整数），用于后续的 epoll_ctl 和 epoll_wait 操作。如果出错返回-1，并设置errno

注册到epoll中的事件是同时组织在一颗红黑树以及一个双向链表上的。可以把一个事件看作是一个节点，epoll_ctl() 添加、修改和删除事件本质上是在红黑树上做添加、修改和删除节点操作，这个复杂度是 $O(\log n)$，n 为所有监听事件总数。当fd就绪时，内核通过回调机制将该事件对应的节点加入到就绪链表上。正因为epoll主动维护就绪列表，所以用户可以直接拿到可用的 fd，无需全量遍历判断。注意就绪链表在内核空间，所以需要 epoll_wait() 将其拷贝到用户空间，也就是拷贝到用户定义的 struct epoll_event[] 数组里。

所以 epoll_ctl() 的时间复杂度是 $O(\log n)$ ，epoll_wait() 的时间复杂度是 $O(1)$ 。

通过对比 epoll 和 select 使用模式的代码（用户空间）可以得出epoll优于select的一个原因： epoll主动维护一个就绪事件链表，用户可以直接得到就绪事件集合。select的就绪事件和非就绪事件都混合存放在一个位数组里，用户无法区分，每次处理就绪事件只能全量遍历整个事件数组。

内核空间实现区别

epoll事件驱动回调

该机制的具体流程为：

  1. 回调函数注册：通过 epoll_ctl(EPOLL_CTL_ADD) 添加 fd 时，内核会为该 fd 的等待队列注册一个回调函数 ep_poll_callback。回调函数与 fd 的底层设备（如网卡、磁盘）关联。(ep_poll_callback是内核或者驱动开发者写的，对于一般使用使用系统API编程的程序员来说是透明的)
  2. 事件触发：当 fd 状态发生变化（如数据到达、缓冲区可写），设备驱动会触发中断，内核调用该 fd 的回调函数。
  3. 就绪队列更新：回调函数将 fd 添加到 epoll 的就绪链表上，并唤醒等待的进程（被 epoll_wait 阻塞的进程）。
  4. 用户获取事件：epoll_wait() 直接从就绪链表拷贝事件 fd 拷贝到用户空间，无需遍历所有的 fd。

// 内核源码示例（简化版）
static int ep_poll_callback(wait_queue_entry_t *wait, ...) {
    struct epitem *epi = ep_item_from_wait(wait);
    list_add_tail(&epi->rdllink, &ep->rdllist);  // 加入就绪队列
    wake_up_locked(&ep->wq);                     // 唤醒进程
}

算法复杂度分析：因为仅有活跃的 fd 会触发回调，不受监控的 fd 总量影响，所以时间复杂度可以看作是 O(1)

select事件驱动轮询

select 也有事件触发机制，比如如果某个被监控的 fd 状态发生变化（数据到达等），内核会立即唤醒 select，但是因为 select没有对每个 fd 注册对应的回调函数，所以 select 只能通过轮询所有 fd（其实就是遍历全量 fd 数组）来检查所有 fd 的状态，然后根据结果决定是否阻塞或返回。

select 轮询的含义既包含周期性定时轮询，也包含单次检查所有 fd 的状态的含义。然后根据结果决定是否阻塞或返回。但是特殊情况下有可能让CPU占用100%。

select 的最后一个参数 timeout 直接控制轮询行为。select 的轮询间隔由 timeout 参数显式定义，但实际执行受内核调度影响：

struct timeval {
    long tv_sec;   // 秒
    long tv_usec;  // 微秒（1秒=1,000,000微秒）
};
  • timeout {0, 0}：立即返回（非阻塞模式，检查一次后无论有无就绪事件都返回），如果直接循环调用会出现100%占满CPU的问题。
  • timeout {>0, >=0}：阻塞等待，最长持续指定时间（期间内核会轮询fd状态）。
  • timeout = NULL：无限阻塞，直到至少一个fd就绪或信号中断。

struct timeval timeout = {1, 0};  // 1秒超时
while (1) {
    int ret = select(nfds, &readfds, NULL, NULL, &timeout);
    if (ret > 0) {
        // 处理就绪fd
    }
    /*
      select 可能会修改 timeout 参数（剩余时间），循环调用时应每次重置。
      如果不重置，当 timeout 被修改为0时，会出现CPU 100% 占用问题。
    */
    timeout.tv_sec = 1;  // 每次循环重置超时
}
