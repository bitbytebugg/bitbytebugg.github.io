freetype依赖

  1. 用VS2022打开freetype.sln，如果本机上没有freetype需要的VS版本和SDK版本，就选择Upgrade选项
  2. 修改项目的属性：
  3. 要求输出目标名字必须是freetype，不能带版本号。dll和lib要分别配置编译一次。 这里要将Runtime Library的/MD改成/MT。configure脚本里会构建一个简单的小程序编译测试freetype依赖是否ok，如果这里不选择/MT选项，那configure脚本就会错误提示需要64位的freetype库，其实这和64位没有关系，是测试编译的链接阶段出现了错误，在/MD选项下，freetype.dll和freetype.lib里对VS依赖的一些符号定义没有链接进自身，测试编译时就会出现freetype.xxx里有未定义符号的错误。
  4. 将编译得到的freetype.dll和freetype.lib放到freetype源码根目录下的lib目录下，然后configure时–with-freetype选项指向freetype的源码根目录就可以了。

OpenJDK构建过程

OpenJDK在Windows上要求cygwin环境，按理说是可以直接使用GNU Make构建的，但是发现有一部分makefile是用nmake的语法（例如文件里使用!IF !ELSE这样的关键字）。所以OpenJDK在Windows平台上使用了GNU Make和NMake两种命令来构建。

hotspot主要目录

hotspot/src是其源代码的主体目录，包含：

  • cpu: 此目录包含一些依赖具体处理器架构的代码
  • os: 此目录下包含一些依赖操作系统的代码
  • os_cpu: 此目录下包含一些依赖操作系统和处理器架构的代码，如linux_x86
  • share: 此目录包含了独立于操作系统和处理器架构的代码，尤其是vm子目录，这个子目录中包含的内容比较多，hotspot主要的一些功能模块都包含在这个子目录里

想象一下我们写代码的环境： Java -> JNI -> 虚拟机实现代码，越往下越接近jvm底层实现，在JNI这一层及其上层一般就是Java开发者写代码的环境，而虚拟机实现代码是需要对Java开发者实现隔离，让他们不需要搞清楚jvm的实现细节就可以方便的写代码。在JVM实现里对象和类分别是用oops和klass类型来表示的，当数据从JVM内部传递到JNI一层时，oops和klass就会转换成这些空类的指针。这就要求这些数据只能在JVM内部处理，因为是空类吗，没有字段和方法，JNI的使用者无法修改他们的。既方便了Java开发者，也保证了JVM内部的安全性。

Klass体系

klass类型用于描述Java类型信息，一个Java类型有一个klass的实例。分成三类 InstanceKlass 和 ArrayKlass以及其他。

_layout_helper描述该对象内存大小：

  • 对于InstanceKlass，就是实例大小（字节为单位）。
  • 对于ArrayKlass，分成4部分，每个部分一个字节
    • hsz记录数组header size（单位字节），也是第一个元素的偏移位置；
    • ebt记录元素的BasicType
    • esz记录元素的个数为2^esz个

_super, _subklass, _next_sibling 三个字段维护继承体系：

  • 由于Java是单继承，所以_super指向唯一的父类的Klass指针
  • _subklass指向其第一个子类
  • _next_sibling指向其父类的所有子类中的下一个兄弟
    • 所以可以通过调用 _subklass->next_sibling()->next_sibling()->…. 获取其所有子类

Hotspot的运行模式

可以通过-version参数查看运行的是哪种模式。

  • 解释模式：可以用 -Xint 指定。在解释模式下又有两种解释器：C++解释器（CppInterpreter）和模板解释器 （TemplateInterpreter）。默认是使用后者。在构建时如果CC_INTERP宏有定义，那么就会指定C++解释器。
    • C++解释器完全是利用软件循环和case-swicth形式解释执行字节码，速度较慢但移植性好。
    • 模板解释器是默认使用的解释器，它为每种字节码指令都生成了汇编代码，使用机器栈和寄存器，速度比C++解释器快10倍，OpenJDK的文档 也指出了这种设计的优势。
  • 编译模式：可以用-Xcomp 指定。Hotspot里的编译器就是我们所说的JIT（即时编译器），内部由两种编译器实现：
    • C1：简单快速的编译器，适用于执行时间较短或者启动性能有要求的程序，例如GUI程序。也被称为Client Compiler。
    • C2：为长期运行的服务端程序做性能优化的编译器，适用于执行时间较长或对峰值性能有要求的程序，也被称为Server Compiler。
  • 混合模式：可以用-Xmixed 指定。一般情况下，我们的代码一开始一般都是由解释器执行，但是当虚拟机发现某个方法或者代码块的运行特别频繁的时候，就会认为这些代码是热点代码。为了提高热点代码的执行效率，会用即时编译器（也就是JIT）把这些热点代码编译成与本地平台相关的机器码，并进行各层次的优化。

TOS缓存优化

这个术语又叫做 Top of Stack Caching。就是在一个栈的栈顶（或附近）元素频繁访问的假设下，对栈顶（或附近）的元素进行缓存的一种优化方式。中文翻译的话通常叫做“栈顶缓存”。

作为参考，可以阅读Anton Ertl发表的论文中Stack caching for interpreters、Implementation of Stack-Based Languages on Register Machines（这篇特别值得一读），和其它一些与“stack caching”相关的论文。

Hotspot VM里的解释器也有使用top-of-stack caching, 简称TOSCA. 在Hotspot VM里看到TOS的话就是top-of-stack的意思，多半都跟TOSCA有关系。

Stack caching有许多可能的变种。有两个指标值得关注： 1、缓存了栈顶附近的多少个元素？如果缓存了n个元素，那么就叫n-TOS caching； 2、缓存带有多少种“状态”？如果有n种状态那么就叫n-state TOS caching。 这俩概念不要弄混了。特别是n-state的概念。

这个优化技术要先从JVM的栈式指令集讲起。由于JVM设计之初就考虑跨平台，而不同CPU架构下如果使用基于寄存器的指令集架构，可能难以实现，所以选择了栈式指令集架构。基于栈的指令级架构相对于基于寄存器的指令集架构优点是跨平台，指令集小，编译器容易实现，每个指令都是相对于栈顶栈帧和栈顶栈帧操作数栈的操作，缺点是性能下降，实现同样的功能需要更多指令。

基于栈的指令集中的一个核心组件就是操作数栈，每个字节码指令取操作数要从操作数栈中取（意味着可能需要压栈并出栈），如果指令有结果需要存储，也需要放到操作数栈顶（可能需要压栈）。所以基于栈式的指令集架构的虚拟机虽然使用的零地址指令更加紧凑，但是单完成一项操作的时候必然需要使用更多的入栈和出栈动作，而这个操作数栈也是存在于内存中，这就意味着将需要更多的内存读写次数，频繁的执行内存读/写操作必然会影响执行速度，为了解决这一问题栈顶缓存技术（Top-of-Stack Caching）便应运而生了。它的核心原理就是将栈顶元素（或者栈顶附近的元素）缓存到物理CPU的寄存器中，以此降低对内存读写的次数，提升执行引擎的执行效率。
