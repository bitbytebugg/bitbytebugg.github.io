Java Socket实现

Mar 01, 2025
2 minute read

在 Linux（或其他操作系统）中，套接字（Socket）本身并不区分客户端和服务端，它们的本质都是通过相同的系统调用（如 socket()）创建的文件描述符。客户端和服务端的角色差异是通过 使用方式 和 通信流程 决定的。

为什么Linux上套接字本身不区分客户端和服务端角色？

  • 灵活性：同一个套接字可以动态切换角色（例如 P2P 应用中，一个套接字可能先作为客户端连接，再作为服务端监听）。
  • 协议无关性：TCP/UDP/Unix Domain Socket 等协议共用相同的套接字接口。
  • 复用性：客户端和服务端的核心操作（如 send()/recv()）完全一致。

在 Java 中，套接字（Socket）区分客户端和服务端（即 Socket 和 ServerSocket 类），主要是为了 简化网络编程模型、明确角色分工，并提供 更高层次的抽象。这种设计并非底层套接字本身的限制，而是 Java 对操作系统原生套接字 API 的封装和优化。

SocketAddress

该类表示一个未关联协议的套接字地址。作为抽象类，它需要通过特定于协议的实现来进行子类化。它提供一个不可变对象被Socket用于绑定、连接操作，或作为返回值使用。

它最常用的子类是 InetSocketAddress（用于 IP 套接字地址，包括TCP和UDP）。

////// jdk-25%2B17/src/java.base/unix/native/libnio/ch/Net.c

JNIEXPORT void JNICALL
Java_sun_nio_ch_Net_bind0(JNIEnv *env, jclass clazz, jobject fdo, jboolean preferIPv6,
                          jboolean useExclBind, jobject iao, int port)
{
    SOCKETADDRESS sa;
    int sa_len = 0;
    int rv = 0;

    if (NET_InetAddressToSockaddr(env, iao, port, &sa, &sa_len,
                                  preferIPv6) != 0) {
        return;
    }
    // 关键逻辑
    rv = NET_Bind(fdval(env, fdo), &sa, sa_len);
    if (rv != 0) {
        handleSocketError(env, errno);
    }
}

////// jdk-25%2B17/src/java.base/unix/native/libnet/net_util_md.c
int NET_Bind(int fd, SOCKETADDRESS *sa, int len)
{
    int rv;

#ifdef __linux__
    /* 如果绑定的 IPv4 地址属于 127.0.0.0/8 网段（即回环地址范围），但不是标准的 127.0.0.1，则直接拒绝绑定。这是 Linux 内核的历史遗留问题
     */
    if (sa->sa.sa_family == AF_INET) {
        if ((ntohl(sa->sa4.sin_addr.s_addr) & 0x7f0000ff) == 0x7f0000ff) {
            errno = EADDRNOTAVAIL;
            return -1;
        }
    }
#endif
    // 调用系统函数 bind() 将套接字绑定到指定地址。
    rv = bind(fd, &sa->sa, len);
    // 返回 bind() 的结果（成功为 0，失败为 -1）。
    return rv;
}

客户端connect0方法

Socket::connect 最终会调用 sun.nio.ch.Net::connect0 方法，它也是一个static native方法，它的C++代码实现在 Net.c 中，最终调用Linux上的connect方法。

Socket和ServerSocket的本质

这二者的本质就是持有一个文件描述符（对应OS创建的套接字），其 bind/connect/accpet等方法都映射到OS的对应方法上。
