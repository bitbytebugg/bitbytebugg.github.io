print命令

查看历史变量

GDB会记录之前打印的值，通过 $n 引用：

2

(gdb) p 10 + 20    # 输出: $1 = 30
(gdb) p $1 + 5     # 输出: $2 = 35

examine命令

简写 x，用于查看指定内存地址内容。

基本语法：x/[数量][格式][单位] 地址：

  • 数量：要显示的内存单元数
  • 格式：
    • x：十六进制
    • d：十进制
    • i：指令（反汇编）
    • a：地址
  • 单位：每个内存单元是多少个字节。数量 x 单位就是总共显示的内存字节个数。
    • b：字节（byte）
    • h：半字（2 字节）
    • w：字（4 字节，默认）
    • g：双字（8 字节）

注意默认显示4个字节，继续回车（不输入命令）会显示接下来的新的4个字节，这个过程可以一直重复。

backtrace命令

简写 bt。

实现原理

理解 backtrace 的关键是理解栈帧（Stack Frame 也称为活动记录 Activation Record）。每次一个函数被调用时，都会在程序的调用栈（Call Stack）上压入一个新的栈帧。

一个简化版的栈帧链看起来是这样的：

1
2
3
4
5
6
7

[ Current Frame (funcC) ]  <- RBP/EBP 寄存器指向这里(Current Frame Pointer)
    | Return Address (in funcB)
    | Previous FP  --> [ Frame (funcB) ]
                           | Return Address (in funcA)
                           | Previous FP  --> [ Frame (funcA) ]
                                                  | ... ...
                                                  | Previous Frame Pointer ......

GDB的backtrace命令正是通过遍历上述的栈帧链来实现的，这个过程大致如下：

  • 获取当前帧指针 (Frame Pointer)：GDB 首先会读取当前线程的 CPU 寄存器，特别是 FP (Frame Pointer) 寄存器。在 x86-64 架构上，这通常是 RBP 寄存器；在 ARM 架构上，是 R11 或 R7 寄存器。这个指针指向当前函数栈帧的底部。
  • 获取当前指令指针 (Instruction Pointer)：读取 IP (Instruction Pointer) 寄存器（x86-64 上是 RIP，ARM 上是 R15/PC）。这告诉了 GDB 当前正在执行的是哪个函数中的哪条指令。
  • 解析当前函数：GDB 拿着 RIP 的值，去查询程序的调试信息（如 DWARF 格式）。调试信息像一个“地图”，记录了地址和函数名、源代码行号的对应关系。通过查找，GDB 就能确定 RIP 所在的函数，这就是 backtrace 的第一行 #0。
  • 遍历栈帧链：
    • 在当前栈帧中，FP 指向的位置附近就存储着上一个栈帧的指针和返回地址。
    • GDB 遵循平台特定的应用程序二进制接口 (ABI) 规则，从当前 FP 指向的内存中取出“上一个 FP”的值和“返回地址”的值。
    • 返回地址：指向调用函数（caller）中的 call 指令之后的下一条指令。GDB 用这个地址去查询调试信息，找到调用它的函数名和行号，形成 backtrace 的下一行 #1。
    • 上一个 FP：成为了新的当前 FP，用于查找再上一层的栈帧。
    • 这个过程循环往复（#2, #3…），直到到达最外层的 main 函数，它的上一个 FP 通常是 NULL，或者遇到了线程栈的边界，遍历就此结束。

调试信息的关键作用

你可能会注意到，仅仅有栈帧遍历只能得到一系列的内存地址。而我们看到的却是清晰的函数名（如 main, foo, bar）和行号（如 main.c:12）。这完全归功于编译时生成的调试信息（通常使用 -g 编译器选项）。这些信息（在 Linux 上通常是 DWARF 格式）存储在可执行文件或独立的调试文件中。它建立了：

  • 内存地址 <-> 函数名的映射
  • 内存地址 <-> 源代码文件和行号的映射 没有调试信息，GDB 只能显示原始地址（如 0x0000555555555169），或者通过反汇编尝试猜测函数名，效果会大打折扣。

特殊情况与增强策略

实际的实现比上述描述更复杂，需要处理各种情况：

  • 帧指针省略 (Frame Pointer Omission - FPO)：
    • 编译器优化（如 -O2, -fomit-frame-pointer）会省略使用 RBP/EBP 作为帧指针，将其用作通用寄存器以提升性能。这会破坏传统的栈帧链。
    • 解决方案：在这种情况下，GDB 会依赖调试信息中的展开信息 (DWARF Unwind Info)。这些信息提供了如何在任何指令点找到上一个栈帧的明确规则。GDB 使用“虚拟展开表”来一步步回溯，即使没有帧指针也能工作。
  • 信号处理函数 (Signal Handlers)：当程序在信号处理函数中中断时，调用栈会包含从正常执行流到信号处理函数的跳转。GDB 能够识别这种特殊情况并将其正确地显示在 backtrace 中。
  • 内联函数 (Inline Functions)：被内联的函数没有自己独立的栈帧。GDB 利用调试信息可以识别出当前地址实际上是一个内联函数，并同时显示内联函数和其外部函数的信息。
  • 不同架构：不同的 CPU 架构（x86, ARM, RISC-V, etc.）有不同的调用约定（Calling Convention）和 ABI。GDB 的实现包含了针对各种架构的特定代码，以正确地解读它们的栈布局。

frame命令

简写 f/fr ，frame命令常常配合backtrace一起使用。backtrack会列出各个stack frame的编号，每行开头紧跟着#后面的数字n。然后将该数字输入命令 frame {n}就可以查看具体栈帧的内容了。

info命令

查看虚表 

1

info vtbl [对象实例名字/对象指针]

查看源代码对应机器指令的内存地址

可以使用info line命令来查看源代码在内存中的地址。info line后面可以跟“行号”，“函数名”，“文件名:行号”，“文件名:函数名”，这个命令会打印出所指定的源码在运行时的内存地址。

查看寄存器

  • info register 显示所有的寄存器（会对照显示16进制的值和10进制的值）
  • info register 寄存器名字（eax，ebx，esp）

list命令

  • list 查看当前后前后各5行代码的内容，当前行算作第6行，总共展示10行。
  • set listsize 设置一次显示代码的行数
  • show listsize 显示listsize的配置
  • list可以跟的参数：
    • linenum 行号
    • beginlinenum, endlinenum 开始行号到结束行号
    • beginlinenum, 开始行号到当前行号
    • , endlinenum 当前行号到结束行号
    • +offset 当前行号的正偏移
    • -offset 当前行号的负偏移
    • filename:linenum 哪个文件的哪一行
    • function 函数名
    • filename:function 哪个文件中的哪个函数

disassemble命令

  • 后面可以跟的参数：
    • 函数名：反汇编整个函数
    • 地址：反汇编包含给定地址的整个函数
    • 起始地址, 结束地址：反汇编起始地址到结束地址中的内容
    • 函数名, +length：反汇编函数开始位置到长度为length个字节部分的内存
    • 地址, +length：反汇编给定内存地址到长度为length个字节部分的内存
    • 地址 -offset,+offset：反汇编地址前后指定偏移量内的代码，注意如果这段范围没有有效的指令代码部分不会被显示。
  • 后面可以指定的选项：
    • /m 其他参数：/m选项会显示与反汇编指令相对应的源代码行。
    • /r 显示所有反汇编指令的原始字节值（也就是把指令的机器码显示出来）
