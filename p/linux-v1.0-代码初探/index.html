Linux v1.0 代码初探

Sep 20, 2024
阅读时长: 9 分钟

编译环境准备

从 这里 下载v1.0版本的内核代码，该版本代码sha1为6bdcbdb4d42c3bec7cbd7e6cc50e7c378acf4faa。看发布时间，Linux的第一个版本是在1994年发布的。

早期的Linux是在Minix上开发的，使用的编译工具也是移植到Minix上的GCC编译器。后续等到Linux版本迭代的越来越成熟后，开源社区将GCC移植到Linux上，此时才支持在旧版本Linux上开发新版本的Linux。

配置内核

通过执行 make config 命令来做构建之前的配置。要求配置的选项在 config.in 中，通过执行Configure shell脚本文件读取 config.in 生成构建所需要的配置文件 .config。.config文件中的内容其实是一些Shell变量定义，这些变量会被Makefile文件include进去，影响Makefile中的构建逻辑。

通过执行 make dep 命令来正确设置所有的依赖。这里的依赖是指编译上的依赖。该命令会生成文件 .depend ，其内容是按照Makefile的目标依赖规则来组织的，其实这部分和在Makefile硬编码指定编译目标的依赖也是可以实现的，只不过因为编译目标的依赖项太多，这种方法比起硬编码的实现方式更加自动化，更加方便。比如 init/main.o 的依赖内容如下：

编译内核

通过执行 make zImage 命令就可以创建一个压缩的内核映像了。由于当时Linux内核编译使用的是非常古老的gcc，所以如果直接在我的实验环境Ubuntu-20.04 with gcc-9.4中编译会失败。一个解决方法是搭建出当时开发Linux内核所用到的Minix操作环境，这个的实现难度很大，一方面在于不清楚当时Minix的版本号，另一方面VMWare和VirtualBox在引导古早的操作系统时会出现问题。另一个解决办法就是修改Linux v1.0的源代码以适应现代编译器。

修改编译参数

  • 由于我使用的操作系统是64位环境，所以编译工具中要增加相关参数，保证生成目标是32位的：
    • 汇编器AS增加参数 --32
    • 链接器LD增加参数 -melf_i386
  • 编译器HOSTCC用于生成构建内核的辅助程序，其执行环境是在Host主机上，这里选择 clang-18 替换原有的 gcc 再增加相关参数：
    • -std=gnu90 使用 gnu C90的语法
    • -m32 生成32位程序
    • -I/home/raymond/workspace/linux/linux-v1.0/include 指定头文件位置
  • 编译器CC用于构建内核映像，其执行环境是在Target主机上，这里选择 clang-18 替换原有的 gcc 再增加相关参数：
    • -std=gnu90
    • -m32
    • –sysroot=/home/raymond/workspace/linux/linux-v1.0/include 使用该选项后，编译器会做如下映射：
      • 默认搜索路径 /usr/include => 实际搜索路径 /home/raymond/workspace/linux/linux-v1.0/include
    • -I/home/raymond/workspace/linux/linux-v1.0/include 指定头文件位置
    • -Wno-int-conversion 忽略相关编译警告
  • CPP预处理器增加参数：
    • -I/home/raymond/workspace/linux/linux-v1.0/include
    • –sysroot=/home/raymond/workspace/linux/linux-v1.0/include

处理链接时找不到符号问题

Linux-v1.0使用的链接器中有一个非常奇葩的规则，如果链接的目标中既有汇编目标文件（.s文件通过汇编器生成）还有编译目标文件（.c文件通过编译器生成），如果这两个目标文件中的全局符号存在相互引用的情况时：

  • 1.汇编目标文件中定义一个全局变量/函数，可以在C目标文件里使用
  • 2.C目标文件里定义一个全局变量/函数，可以在汇编目标文件里使用 这时链接器会要求汇编目标文件使用的符号前加 _ ，而C目标文件中的符号保持不变。现代链接器已经没有这种不合理的规则了，但是如果不修改代码，现代编译器会报无法找到符号定义的错误。我的解决办法是将交叉使用的符号在所有的汇编代码里 _ 前缀去掉。这个改动非常大，至少有几百处位置修改，但是除此之外没有更好的办法了。

预处理器语法修改

Linux-v1.0中C代码会使用 #ifdef 0 来隐藏不需要被编译的代码，但是现代编译器不支持这种语法，直接改成 #ifdef NO_DEF_REPLACE_0 即可。

extern inline 改为 static inline

在头文件中定义的 extern inline 函数，在现代编译器下会报多重定义的问题，将 extern 关键字改为 static 关键字可以通过编译。这是因为static限制符号只在文件内部可见。

内联汇编语法

GCC 内联汇编是一种在 C/C++ 代码中直接嵌入汇编指令的强大功能，GCC 使用 AT&T 语法。基本的语法结构如下：

1
2
3
4
5
6

asm [volatile] (
    "汇编指令模板"
    : 输出操作数列表      /* 可选，形式为： "操作数约束字符1" (C环境变量1), "操作数约束字符2" (C环境变量2), ... */
    : 输入操作数列表      /* 可选，形式同上 */
    : 被破坏的寄存器列表  /* 可选，形式为："寄存器1名字", "寄存器2名字", ...*/
);
  • asm：关键字，也可以用 __asm__，后者更常见，因为可以避免与代码中其他名为 asm 的标识符冲突。
  • volatile：可选关键字，也可以用 __volatile__，理由同上。它告诉编译器不要优化这段汇编代码，防止被意外删除或移动。在涉及内存操作或硬件I/O时，几乎总是需要它。
  • 括号内包含四个主要部分，用冒号分隔。
    • 汇编指令模板：这是一个字符串，包含你的汇编指令。其中可以包含占位符，这些占位符会在编译时被具体的操作数替换。
      • 占位符语法：%0, %1, %2, … 按操作数出现的顺序编号（先编码输出操作数编号，后编码输入操作数编号）。
      • 寄存器名前加两个百分号 %%（如 %%eax），以区别于占位符（如 %0）。
      • 使用 \n 或 \t 来格式化汇编代码。
    • 输入/输出操作数约束：也就是上述语法中的输出操作数列表和输入操作数列表中的字符串部分中的大小写字母。它告诉编译器如何为C语言操作数分配寄存器或内存。
      • 约束 含义
      • r 让编译器选择任何一个通用寄存器
      • a 使用 %eax, %ax, %al
      • b 使用 %ebx, %bx, %bl
      • c 使用 %ecx, %cx, %cl
      • d 使用 %edx, %dx, %dl
      • S 使用 %esi, %si
      • D 使用 %edi, %di
      • m 使用内存地址（而不是寄存器）
      • i 立即数（编译时常量）
      • g 寄存器、内存或立即数（任何通用类型）
    • 输出操作数修饰符：也就是上述语法中输出操作数列表和输入操作数列表中的字符串部分中的特殊符号。
      • 修饰符 含义
      • = 只写（通常用于输出操作数）
      • + 读写（操作数既是输入又是输出）
      • & 早期破坏寄存器（不能与输入操作数共用寄存器）
    • 被破坏的寄存器列表（GCC手册中叫做Clobber List）：这个部分告诉编译器我们在汇编代码中显式修改了哪些寄存器（除了通过输出操作数隐式修改的之外）。
      • 寄存器名：“eax”, “ebx” 等
      • “memory”：表示我们修改了内存（不仅仅是输出操作数指向的内存），编译器在执行汇编代码前会将所有内存值写回，并在执行后重新读取。
      • “cc”：表示我们修改了条件码寄存器（EFLAGS）。
  • 扩展汇编模板（Named Operands）：为了避免手动计算 %0, %1 的麻烦，GCC 支持命名操作数，这样代码更易读和维护。如下示例：

int src = 10, dst;

asm volatile (
    "movl %[input], %%eax\n"
    "movl %%eax, %[output]"
    : [output] "=r" (dst)   // 输出操作数命名为 output
    : [input] "r" (src)     // 输入操作数命名为 input
    : "%eax"
);

内联汇编代码修改

Linux-v1.0中C代码会使用内联汇编，但是当时的内联汇编语法和现代编译器支持的内联汇编语法有些区别，差异在于现代gcc要求如果被破坏的寄存器在输入操作数中出现，那就需要写到被破坏的寄存器列表里。比如这段代码：

1
2
3

__asm__ __volatile__("rep ; movsl": \
	:"c" (((unsigned long) size) >> 2),"S" (from),"D" (to) \
	:"cx","di","si")

因为 cx, di, si在输入操作数列表中都出现了，所以最后的Clobber List中要置为空。

使用链接器脚本控制内核映像格式

Linux-v1.0代码中没有加载器的概念，都是硬盘上的可执行文件直接映射到内存中的一段区域，为了适应这种限制，只能用链接器脚本(*.lds)让链接器把.text段空出一个特定的偏移，并且.text, .data, .rodata, .bss 段的file offset和虚拟地址保持一致。

比如生成zSystem的控制脚本为：

SECTIONS {
    . = 0x1000; 
    .text : { *(.text) }
    .rodata : { *(.rodata) *(.kernel_zip_data)}
    .data : { *(.data) }
    PROVIDE(edata = .);
    .bss : { *(.bss) }
    PROVIDE(end = .);
}

构建映像中的辅助工具

xtract.c

zBoot/xtract.c是一个可执行程序，用于从 tools/zSystem（从stdin中读取） 中的 .text .data .bss 段数据提取出来，输出到stdout。

由于早期Linux都是用Minix开发的，随着Linux内核逐渐完善后才自举开发（在Linux上开发Linux）。猜测在Linux 1.0 构建二进制可执行文件格式应该是 a.out 格式，从 xtract.c 中包含的头文件 <a.out.h> 可以验证这一推测，但是现代构建工具已经不支持 a.out 的，生成的是elf格式。所以 xtract.c 的逻辑需要改写。

xtract.c 从stdin里读取 ELF 格式的输入文件 tools/zSystem，从中提取代码段和数据段，生成新的 a.out 格式的文件输出到 stdout。

通过 xtract.c 代码逻辑猜测 a.out 格式如下：

  • 最开始的1024字节就是 GCC_HEADER，这1024个字节的最开始部分包含了 a.out.h 中的结构体 struct exec内容。
  • 1024字节后的部分就是 .text段
  • 然后紧接着 .data 段
  • 然后紧接着是符号表。xtract.c 中通过符号表开始位置减去GCC_HEADER获取 .text段和 .data 段总共的大小 N_SYMOFF(*ex) - GCC_HEADER

piggyback.c

xtract.c 提取了 .text段和 .data 段后的 a.out 文件经过gzip压缩后从stdin传递给 piggyback.c 程序。该程序的作用是模拟链接器将 stdin 中输入的内容链接到一个可重定位的 piggy.o 文件中（从stdout输出）。piggy.o 其中有一个全局符号 _input_data，该符号是 piggyback.c 从stdin中读取到数据写入的起始位置，另外还有一个全局符号 _input_len 该符号是记录的是piggyback.c 从stdin中读取到数据的长度。

build.c 工具

将 boot/bootsect, boot/setup 和 zBoot/zSystem 依次写入zImage里。zImage就是生成的内核镜像。zBoot/zSystem会做strip操作，只写入 .text 段 和 .data段，减少映像文件大小。

使用qemu模拟执行内核映像

一开始没有指定machine参数，发现boot阶段有一段复制磁盘的逻辑无法生效。如果发现qemu不停reboot，那一般就是BIOS环境不支持，考虑到Linux-v1.0是在1994年发布的，所以用-machine pc-1.0参数指定一个较古老的机器环境。

经过多次探索，最终使用如下命令启动qemu的模拟器，能成功引导内核映像：

# -boot a 表示从第一块软盘启动
# -fda 指定软盘映像的文件
# -d 所选择的日志会直接打印到terminal上
# -s 开启gdb调试端口
# -S 启动后立刻停止
qemu-system-i386 -machine pc-1.0 -boot a -fda zImage -d in_asm,guest_errors -s -S
# -machine help 查看更多机器类型
