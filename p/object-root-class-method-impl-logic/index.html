Object根类方法实现逻辑

Feb 10, 2025
阅读时长: 16 分钟

Object类定义的方法非常重要，因为其他对象都是继承自该类，会默认继承这些方法的实现。所以Object类定义的方法几乎影响一切对象的行为。虽然native成员方法可以被重写为java成员方法，但是性能上可能会达不到要求，而且Java程序员几乎是不会去写C++本地方法的，所以很少有Java代码会出现重写Object父类的本地成员方法的情况。可以把Object类的native成员方法看作是Java世界中所有对象的典型的模板化的逻辑。

hashCode方法

Java代码实现：

1
2

////// Object.java
public native int hashCode();

在JDK的注释中对hashCode方法有如下约定：

  • 在 Java 应用程序的一次执行期间，对同一对象多次调用 hashCode 方法时，只要对象在 equals 比较中使用的信息未被修改，该方法必须始终返回相同的整数。该整数在不同应用程序执行间不必保持一致。
  • 如果两个对象通过 equals(Object) 方法判断相等，那么对这两个对象分别调用 hashCode 方法必须产生相同的整数结果。
  • 如果两个对象通过 equals(Object) 方法判断不相等，不要求对这两个对象调用 hashCode 方法必须产生不同的整数结果。但程序员应当意识到，为不相等的对象生成不同的哈希码可以提高哈希表的性能。
  • 在合理可行的范围内，Object 类定义的 hashCode 方法确实会为不同的对象返回不同的整数。（典型的实现方式是将对象的内部地址转换为整数，但 Java编程语言并不强制要求采用这种实现技术。）

在 Java 中，hashCode() 方法的主要作用就是为了配合哈希表使用的。像 Java 中的 HashSet、Hashtable（注意是小写的 t）、HashMap 都是基于哈希表的具体实现。其中的 HashMap 就是最典型的代表。拿 HashMap 来说吧。当我们要在它里面添加对象时，先调用这个对象的 hashCode() 方法，得到对应的哈希值，然后将哈希值和对象一起放到 HashMap 中。当我们要再添加一个新的对象时：

  1. 获取对象的哈希值；
  2. 和之前已经存在的哈希值进行比较，如果不相等，直接存进去；
  3. 如果有相等的，再调用 equals() 方法进行对象之间的比较，如果相等，不存了；
  4. 如果不等，说明哈希冲突了，增加一个链表，存放新的对象；
  5. 如果链表的长度大于 8，转为红黑树来处理。

我们可以这样描述 hashCode() 和 equals() 之间的关系：

  • 如果两个对象调用 equals() 方法得到的结果为 true，调用 hashCode() 方法得到的结果必定相等；
  • 如果两个对象调用 equals() 方法得到的结果为 false，调用 hashCode() 方法得到的结果不一定不相等；
  • 如果两个对象调用 hashCode() 方法得到的结果不相等，调用 equals() 方法得到的结果必定为 false；
  • 如果两个对象调用 hashCode() 方法得到的结果相等，调用 equals() 方法得到的结果不一定为 true；

////// jdk8-b120/hotspot/src/share/vm/runtime/synchronizer.cpp
intptr_t ObjectSynchronizer::FastHashCode (Thread * Self, oop obj) {
  // 0. 偏向锁情况处理
  if (UseBiasedLocking) {...}
  ObjectMonitor* monitor = NULL;
  markOop temp, test;
  intptr_t hash;
  // 读取对象的 mark word
  markOop mark = ReadStableMark (obj);
  // 1. 无锁状态处理
  if (mark->is_neutral()) {
    hash = mark->hash();
    if (hash) {  // 已有哈希码直接返回
      return hash;
    }
    hash = get_next_hash(Self, obj);  // 没有的情况计算hash值
    temp = mark->copy_set_hash(hash); // hash值写入mark word
    test = (markOop) Atomic::cmpxchg_ptr(temp, obj->mark_addr(), mark);
    if (test == mark) {
      return hash;
    }
  // 2. 重量级锁处理
  } else if (mark->has_monitor()) {
    monitor = mark->monitor();
    temp = monitor->header();
    assert (temp->is_neutral(), "invariant") ;
    hash = temp->hash();
    if (hash) {
      return hash;
    }
  // 3. 轻量级锁处理（当前线程持有）
  } else if (Self->is_lock_owned((address)mark->locker())) {
    temp = mark->displaced_mark_helper();
    assert (temp->is_neutral(), "invariant") ;
    hash = temp->hash();
    if (hash) {
      return hash;
    }
  }

  // 前述路径失败时膨胀为重量级锁
  monitor = ObjectSynchronizer::inflate(Self, obj);
  mark = monitor->header();
  assert (mark->is_neutral(), "invariant") ;
  hash = mark->hash();
  if (hash == 0) {
    hash = get_next_hash(Self, obj);
    temp = mark->copy_set_hash(hash); // merge hash code into header
    assert (temp->is_neutral(), "invariant") ;
    test = (markOop) Atomic::cmpxchg_ptr(temp, monitor, mark);
    if (test != mark) {
      hash = test->hash();
      assert (test->is_neutral(), "invariant") ;
      assert (hash != 0, "Trivial unexpected object/monitor header usage.");
    }
  }
  return hash;
}

总之，已经计算出hash的情况下直接返回（存储地方在哪里？），否则调用 get_next_hash 方法计算hash。

static inline intptr_t get_next_hash(Thread* current, oop obj) {
  intptr_t value = 0;
// hashCode变量是JVM启动时的一个全局参数，可以通过设置它来切换哈希值生成的策略，默认值是5
  if (hashCode == 0) {
    // 调用操作系统 OS 的 random() 方法返回随机数。
    value = os::random();
  } else if (hashCode == 1) {
    intptr_t addr_bits = cast_from_oop<intptr_t>(obj) >> 3;
    value = addr_bits ^ (addr_bits >> 5) ^ GVars.stw_random;
  } else if (hashCode == 2) {
    value = 1;  // 某些情况下的测试
  } else if (hashCode == 3) {
    // 从 0 开始计算哈希值，不是线程安全的，多个线程可能会得到相同的哈希值。
    value = ++GVars.hc_sequence;
  } else if (hashCode == 4) {
    // 与创建对象的内存位置有关，原样输出。
    value = cast_from_oop<intptr_t>(obj);
  } else {
    /* 该策略是默认值，支持多线程，使用了 Marsaglia 的 xor-shift 算法产生伪随机数。
     所谓的 xor-shift 算法，简单来说，看起来就是一个移位寄存器，每次移入的位由寄存器
     中若干位取异或生成。所谓的伪随机数，不是完全随机的，但是真随机生成比较困难，所以
     只要能通过一定的随机数统计检测，就可以当作真随机数来使用。
    */
    unsigned t = current->_hashStateX;
    t ^= (t << 11);
    current->_hashStateX = current->_hashStateY;
    current->_hashStateY = current->_hashStateZ;
    current->_hashStateZ = current->_hashStateW;
    unsigned v = current->_hashStateW;
    v = (v ^ (v >> 19)) ^ (t ^ (t >> 8));
    current->_hashStateW = v;
    value = v;
  }

  value &= markWord::hash_mask;
  if (value == 0) value = 0xBAD;
  assert(value != markWord::no_hash, "invariant");
  return value;
}

可以看到默认策略（hashCode = 5）不再显式混合对象地址，而是完全依赖 Xorshift 的随机性。如果完全依靠xorshift的随机性可能会想到一个问题：重复调用同一个对象的hashCode()会得到不同的返回值吗？答案是不会，因为Java 语言规范对 hashCode() 的强制性约定要求同一对象得到的hashCode()返回值一定相同。

这个不变性保证的实现原理是调用 get_next_hash() 只会计算一次对象hash，后面都会缓存该值。不过缓存的地方非常复杂，会根据加锁状态变化，这是JVM优化策略决定的。

equals方法

这不是一个native方法，也是被继承对象经常重写的方法。

1
2
3
4

////// Object.java
public boolean equals(Object obj) {
    return (this == obj);
}

以下翻译自JDK注释： 判断其他对象是否"等于"当前对象。equals 方法在非空对象引用上实现等价关系：

  • 自反性：对于任何非空引用值 x，x.equals(x) 都应返回 true。
  • 对称性：对于任何非空引用值 x 和 y，当且仅当 y.equals(x) 返回 true 时，x.equals(y) 才应返回 true。
  • 传递性：对于任何非空引用值 x、y 和 z，如果 x.equals(y) 返回 true 且 y.equals(z) 返回 true，那么 x.equals(z) 也应返回 true。
  • 一致性：对于任何非空引用值 x 和 y，在对象上用于 equals 比较的信息未被修改的前提下，多次调用 x.equals(y) 应一致地返回 true 或 false。
  • 非空性：对于任何非空引用值 x，x.equals(null) 都应返回 false。 Object 类的 equals 方法实现了对象上最具区分性的等价关系；即对于任何非空引用值 x 和 y，当且仅当 x 和 y 引用同一对象（x == y 的值为 true）时，该方法才返回 true。

请注意，在重写此方法时，通常需要同时重写 hashCode 方法，以维护 hashCode 方法的通用约定，该约定声明相等的对象必须具有相等的哈希码。

clone方法 

1
2

////// Object.java
protected native Object clone() throws CloneNotSupportedException;

翻译自JDK注释： 创建并返回此对象的副本。“副本"的具体含义取决于对象的类。通常的意图是，对于任何对象 x，表达式：x.clone() != x 将为 true ；且表达式：x.clone().getClass() == x.getClass() 将为 true，但这些并非绝对要求。

按照约定，返回的对象应通过调用 super.clone 获得。如果一个类及其所有超类（Object 除外）都遵守此约定，则将满足 x.clone().getClass() == x.getClass()。

按照约定，此方法返回的对象应独立于被克隆的对象。为实现这种独立性，可能需要在返回前修改 super.clone() 返回对象的一个或多个字段。这通常意味着复制构成被克隆对象内部"深层结构"的任何可变对象，并将对这些对象的引用替换为对副本的引用。如果类仅包含基本类型字段或对不可变对象的引用，则通常不需要修改 super.clone() 返回对象中的任何字段。

Object 类的 clone 方法执行特定的克隆操作。首先，如果此对象的类未实现 Cloneable 接口，则抛出 CloneNotSupportedException。请注意，所有数组都被视为实现了 Cloneable 接口，且数组类型 T[] 的 clone 方法返回类型为 T[]，其中 T 是任何引用或基本类型。否则，此方法会创建此对象类的新实例，并通过赋值的方式完全初始化其所有字段为被克隆对象对应字段的内容；字段内容本身不会被克隆。因此，此方法执行的是此对象的"浅拷贝”，而非"深拷贝"操作。

Object 类本身并未实现 Cloneable 接口，因此在 Object 类实例上调用 clone 方法将在运行时抛出异常。

正因为不能直接调用Object上的clone方法（因为Object没有实现Cloneable接口，直接调会抛出异常），所以上面Java代码里clone方法是被 protected 修饰的，只能被其子类在内部调用。

JVM_ENTRY(jobject, JVM_Clone(JNIEnv* env, jobject handle))
  JVMWrapper("JVM_Clone");
  Handle obj(THREAD, JNIHandles::resolve_non_null(handle));
  const KlassHandle klass (THREAD, obj->klass());
  JvmtiVMObjectAllocEventCollector oam;
  // 1. 检查是否实现了Cloneable接口，没有直接抛出异常
  if (!klass->is_cloneable()) {
    ResourceMark rm(THREAD);
    THROW_MSG_0(vmSymbols::java_lang_CloneNotSupportedException(), klass->external_name());
  }
  // 2. 分配复制对象的内存（二分模型中oop所占用的内存）
  const int size = obj->size();
  oop new_obj = NULL;
  if (obj->is_array()) {
    const int length = ((arrayOop)obj())->length();
    new_obj = CollectedHeap::array_allocate(klass, size, length, CHECK_NULL);
  } else {
    new_obj = CollectedHeap::obj_allocate(klass, size, CHECK_NULL);
  }
  // 3. 执行浅复制。下面的函数可以看作是内存块的复制操作。
  Copy::conjoint_jlongs_atomic((jlong*)obj(), (jlong*)new_obj,
                               (size_t)align_object_size(size) / HeapWordsPerLong);
  // 初始化新对象的Mark Word对象头
  new_obj->init_mark();
  // 内存写屏障设置，确保对象初始化对其他线程可见
  BarrierSet* bs = Universe::heap()->barrier_set();
  bs->write_region(MemRegion((HeapWord*)new_obj, size));
  // 注册finalizer，看起来不是主逻辑
  if (klass->has_finalizer()) {
    assert(obj->is_instance(), "should be instanceOop");
    new_obj = InstanceKlass::register_finalizer(instanceOop(new_obj), CHECK_NULL);
  }
  // 返回浅拷贝的对象
  return JNIHandles::make_local(env, oop(new_obj));
JVM_END

总结Object的clone方法就是对两个Java对象在JVM底层的二分模型中的oop部分数据做内存块的复制。这就是浅拷贝的含义。基本数据类型字段做了独立拷贝，引用数据类型字段则实际是指向同一个对象地址。

wait方法

从Java代码里可以看出这又是一个native方法：

1
2

////// Object.java
public final native void wait(long timeout) throws InterruptedException;

翻译自JDK注释： 使当前线程进入等待状态，直到以下任一情况发生：

  1. 其他线程调用此对象的 notify() 方法
  2. 其他线程调用此对象的 notifyAll() 方法
  3. 指定的等待时间已到

当前线程必须持有该对象的监视器锁。此方法会使当前线程（称为 T）将自己置于该对象的等待集中，并释放对该对象的所有同步声明。线程 T 将被禁用线程调度，并处于休眠状态，直到发生以下四种情况之一：

  • 其他线程调用此对象的 notify 方法，且线程 T 被随机选中唤醒
  • 其他线程调用此对象的 notifyAll 方法
  • 其他线程中断线程 T（通过调用 T.interrupt() ）
  • 指定的实时等待时间已到（允许一定误差）。若 timeout 为零，则不考虑时间因素，线程将一直等待直到被通知

随后线程 T 将从该对象的等待集中移除，并重新启用线程调度。之后它将与其他线程以常规方式竞争对该对象的同步权；一旦重新获得对象的控制权，它对该对象的所有同步声明都将恢复到调用 wait 方法之前的状态 - 即恢复到调用 wait 方法时的状态。随后线程 T 从 wait 方法调用中返回。因此，从 wait 方法返回时，对象和线程 T 的同步状态与调用 wait 方法时完全相同。

线程也可能在没有收到通知、中断或超时的情况下被唤醒，这种现象称为"虚假唤醒"(spurious wakeup)。虽然实践中很少发生，但应用程序必须通过检查本应使线程唤醒的条件来防范这种情况，如果条件不满足则应继续等待。换句话说，wait 操作应始终放在循环中，如下所示：

synchronized (obj) {
    while (<条件不满足>)
        obj.wait(timeout);
    // 执行条件满足后的操作
    ... 
}

（更多信息请参阅 Doug Lea 所著《Java 并发编程（第二版）》（Addison-Wesley，2000）第 3.2.3 节，或 Joshua Bloch 所著《Effective Java 编程语言指南》（Addison-Wesley，2001）第 50 条）

如果当前线程在等待前或等待期间被任何线程中断，则会抛出 InterruptedException。在对象的锁状态按上述描述恢复前，不会抛出此异常。

注意 wait 方法只会释放当前对象的锁，当前线程可能持有的其他对象锁在线程等待期间仍保持锁定状态。

只有持有该对象监视器锁的线程才能调用此方法。关于线程如何成为监视器锁的所有者，请参阅 notify 方法的说明。

void ObjectSynchronizer::wait(Handle obj, jlong millis, TRAPS) {
  // 偏向锁？

    assert(!obj->mark()->has_bias_pattern(), "biases should be revoked by now");

  // 3. 在监视器锁上调用wait方法

// 以下是deepseek基于JDK17代码精简后的核心逻辑
void ObjectMonitor::wait(jlong millis, bool interruptible, TRAPS) {
  Thread* const self = THREAD;
  // 1. 检查当前线程是否持有锁
  if (THREAD != _owner) {
    if (THREAD->is_lock_owned((address)_owner)) {
      _owner = THREAD; // 处理锁重入
    } else {
      THROW_MSG(vmSymbols::java_lang_IllegalMonitorStateException(), 
               "current thread not owner");
    }
  }
  // 2. 创建 WaitNode 并加入等待队列
  WaitNode node(self, millis);
  AddWaiter(&_WaitSet, &node);
  // 3. 释放锁并退出监视器
  exit(true, self);
  // 4. 线程挂起（可响应中断）
  int ret = self->_ParkEvent->park(millis);
  // 5. 唤醒后重新竞争锁
  enter(self);
  // 6. 处理中断
  if (interruptible && self->is_interrupted()) {
    THROW(vmSymbols::java_lang_InterruptedException());
  }
}

////// Object.java
public final native void notify();

翻译自JDK注释： 唤醒在此对象监视器上等待的单个线程。如果有多个线程正在等待该对象，则会选择其中一个线程唤醒。选择是任意的，具体由实现决定。线程通过调用 wait 方法在对象的监视器上等待。

被唤醒的线程将无法继续执行，直到当前线程释放此对象的锁。被唤醒的线程将以常规方式与其他可能正在竞争同步此对象的线程进行竞争；例如，被唤醒的线程在成为下一个锁定此对象的线程时，既不享有可靠的特权，也不会处于不利地位。

只有持有此对象监视器所有权的线程才能调用此方法。线程可以通过以下三种方式之一成为对象监视器的所有者：

  • 执行该对象的同步实例方法（synchronized修饰的实例方法）
  • 执行同步在该对象上的 synchronized 语句块
  • 对于 Class 类型的对象，执行该类的同步静态方法

一个对象的监视器在同一时间只能被一个线程拥有。

// deepseek生成
void ObjectMonitor::notify(TRAPS) {
  // 1. 检查当前线程是否持有锁
  if (THREAD != _owner) {
    if (THREAD->is_lock_owned((address)_owner)) {
      _owner = THREAD; // 处理锁重入情况
    } else {
      THROW_MSG(vmSymbols::java_lang_IllegalMonitorStateException(),
               "current thread not owner");
      return;
    }
  }
  
  // 2. 如果等待集为空则直接返回
  if (_WaitSet == NULL) return;
  
  // 3. 从等待集（LIFO链表）头部取出一个线程
  ObjectWaiter* iterator = _WaitSet;
  _WaitSet = iterator->_next;

  // 4. 根据策略将线程转移到EntryList或cxq
  // 通过-XX:MonitorPolicy设置
  if (Policy == 0) {       // 插入EntryList头部（默认策略）
    iterator->_next = _EntryList;
    _EntryList = iterator;
  } else if (Policy == 1) { // 插入EntryList尾部
    // ...遍历链表到尾部插入
  } else if (Policy == 2) { // 插入cxq头部
    iterator->_next = _cxq;
    _cxq = iterator;
  } else {                 // 插入cxq尾部（Policy == 3）
    // ...遍历cxq链表到尾部插入
  }

  // 5. 设置线程状态标记
  iterator->TState = ObjectWaiter::TS_ENTER;

  // 6. 实际唤醒线程，唤醒线程后在ObjectMonitor::wait()里重新获取锁。
  // 也就是ObjectMonitor::wait 函数返回前的 enter(self) 语句。
  ParkEvent * ev = iterator->_event;
  if (ev != NULL) {
    ev->unpark();
  }
}

////// jdk8-b120/hotspot/src/share/vm/runtime/objectMonitor.cpp
void ObjectMonitor::notifyAll(TRAPS) {
  // 1. 检查当前线程是否是该锁的拥有者
  CHECK_OWNER();
  ObjectWaiter* iterator;
  if (_WaitSet == NULL) {
      TEVENT (Empty-NotifyAll) ;
      return ;
  }

  int Policy = Knob_MoveNotifyee ;
  int Tally = 0 ;
  Thread::SpinAcquire (&_WaitSetLock, "WaitSet - notifyall") ;
  
  // 2. 遍历等待线程队列，唤醒其中的每个线程
  for (;;) {
     iterator = DequeueWaiter () ;
     if (iterator == NULL) break ;
     TEVENT (NotifyAll - Transfer1) ;
     ++Tally ;

     guarantee (iterator->TState == ObjectWaiter::TS_WAIT, "invariant") ;
     guarantee (iterator->_notified == 0, "invariant") ;
     iterator->_notified = 1 ;
     // 通过宏获取当前线程实例对象（类型是JavaThread）
     Thread * Self = THREAD;
     iterator->_notifier_tid = Self->osthread()->thread_id();
     if (Policy != 4) {
        iterator->TState = ObjectWaiter::TS_ENTER ;
     }
     ObjectWaiter * List = _EntryList ;
     // 根据不同策略选择唤醒线程的方式
     if (Policy == 0) {       // prepend to EntryList
         if (List == NULL) {
             iterator->_next = iterator->_prev = NULL ;
             _EntryList = iterator ;
         } else {
             List->_prev = iterator ;
             iterator->_next = List ;
             iterator->_prev = NULL ;
             _EntryList = iterator ;
        }
     } else
     if (Policy == 1) {      // append to EntryList
         if (List == NULL) {
             iterator->_next = iterator->_prev = NULL ;
             _EntryList = iterator ;
         } else {
            // CONSIDER:  finding the tail currently requires a linear-time walk of
            // the EntryList.  We can make tail access constant-time by converting to
            // a CDLL instead of using our current DLL.
            ObjectWaiter * Tail ;
            for (Tail = List ; Tail->_next != NULL ; Tail = Tail->_next) ;
            assert (Tail != NULL && Tail->_next == NULL, "invariant") ;
            Tail->_next = iterator ;
            iterator->_prev = Tail ;
            iterator->_next = NULL ;
        }
     } else
     if (Policy == 2) {      // prepend to cxq
         // prepend to cxq
         iterator->TState = ObjectWaiter::TS_CXQ ;
         for (;;) {
             ObjectWaiter * Front = _cxq ;
             iterator->_next = Front ;
             if (Atomic::cmpxchg_ptr (iterator, &_cxq, Front) == Front) {
                 break ;
             }
         }
     } else
     if (Policy == 3) {      // append to cxq
        iterator->TState = ObjectWaiter::TS_CXQ ;
        for (;;) {
            ObjectWaiter * Tail ;
            Tail = _cxq ;
            if (Tail == NULL) {
                iterator->_next = NULL ;
                if (Atomic::cmpxchg_ptr (iterator, &_cxq, NULL) == NULL) {
                   break ;
                }
            } else {
                while (Tail->_next != NULL) Tail = Tail->_next ;
                Tail->_next = iterator ;
                iterator->_prev = Tail ;
                iterator->_next = NULL ;
                break ;
            }
        }
     } else {
        ParkEvent * ev = iterator->_event ;
        iterator->TState = ObjectWaiter::TS_RUN ;
        OrderAccess::fence() ;
        // 3. 唤醒当前遍历到的线程，被唤醒额线程在ObjectMonitor::wait方法中继续竞争锁。
        ev->unpark() ;
     }

     if (Policy < 4) {
       iterator->wait_reenter_begin(this);
     }
  }  // end for loop

  Thread::SpinRelease (&_WaitSetLock) ;

  if (Tally != 0 && ObjectMonitor::_sync_Notifications != NULL) {
     ObjectMonitor::_sync_Notifications->inc(Tally) ;
  }
}
