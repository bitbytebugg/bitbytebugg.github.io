程序ELF格式布局初探

Jul 19, 2024
4 minute read

实验环境准备

可执行程序文件在磁盘上的存储格式和操作系统相关，在Windows平台上的可执行程序文件是PE格式，在Linux平台上就是ELF格式。EFL全称为Executable and Linkable Format。其实这里表达不是很准确，ELF文件可以细分三种类型：可执行程序、可重定位文件（单个编译单元编译后的产物，未进行链接操作，Linux上的扩展名字一般是 .o）和共享目标文件（也就是动态链接库，在Linux上的扩展名字一般是 .so）

一般学习过计算机专业课都会知道程序包含代码和数据两部分，这里我们通过编译一个最小的C可执行程序，来分析ELF是如何存储代码和数据的：

// main.c

char strarr[] = "xyz";
int g_int = 0x12345678;
char* strp = "abcdefgh";

int main() {

}

我们通过这段gcc命令来编译得到可执行程序：gcc -nostdinc -nostdlib -static --entry main main.c。这里的编译是广义的概念，内部隐藏了链接的操作，而狭义的编译概念不包含链接操作，对应于gcc命令里的 -c 参数。

另外快速解释上面命令中的几个参数的含义：

  • nostdinc：禁止在标准系统目录中搜索头文件
  • nostdlib：不链接标准启动文件和标准库
  • static：生成完全静态链接的可执行文件
  • entry main：指定程序入口点为 main 函数

增加这些参数的目的是简化我们的分析，剔除和我们分析无关的代码。

好了，现在我们得到分析目标 a.out 可执行程序，因为没有通过 -o 参数指定输出文件名字，所以gcc使用默认名称a.out。

size命令查看各个段大小 

1
2
3

size a.out 
   text	   data	    bss	    dec	    hex	filename
    148	     16	      0	    164	     a4	a.out

从命令的输出结果中可以看出代码段(text)在ELF格式中占用了148个字节，数据段(data)在ELF格式中占据了16字节。变量strarr 4个字节 + g_int 4个字节 + strp 9个字节 = 17个字节，怎么和上面size命令的结果对不上呢？注意这里 strp 只是一个字符串指针，在64位系统上指针的大小是8个字节，所以其实 strp 在 data段里只占用8个字节，它指向的内容存放在另外的地方。后面会通过另一个工具readelf来验证这里的逻辑。

readelf查看.data section内容 

1
2
3
4

readelf -x .data a.out 

Hex dump of section '.data':
  0x00404000 78797a00 78563412 00204000 00000000 xyz.xV4.. @.....

输出的内容是从虚拟地址 0x00404000 开始的16个字节的内容，我们可以看到所有全局字段的初始值是连续存储的：

  • 0x78797a00 存储 strarr[]的初始值，注意按照C语言字符串规范，最后以 \0 标记字符串结束位置。
  • 0x78563412 存储 g_int 的初始值，注意按照Little Endian规则存储。
  • 0x00204000 00000000 存储的一个 strp 的64位指针指向的地址（注意Little Endian序）。这个地址的内容就是实际字符串的值。记住这个地址，后面验证过程会出现这个地址。

readelf查看.rodata section内容 

1
2
3
4

readelf -x .rodata a.out 

Hex dump of section '.rodata':
  0x00402000 61626364 65666768 00                abcdefgh.

从输出结果中我们可以看到.rodata section的起始地址就是上面 strp 指向的地址，这是巧合吗？并非如此。后面字符串的内容 0x61626364 0x65666768 0x00 说明这个地址存放的内容就是 strp 指向的字符串内容。

当然这是所有机器指令混合在一起输出的，对于人类阅读而言，非常不直观。这里还有一种更直观的查看代码段的方法：

0:  f3 0f 1e fa             endbr64
4:  55                      push   rbp
5:  48 89 e5                mov    rbp,rsp
8:  b8 00 00 00 00          mov    eax,0x0
d:  5d                      pop    rbp
e:  c3                      ret

objdump直接将每条机器指令分开展示，并且每天机器指令对应的汇编代码也反解出来了。
